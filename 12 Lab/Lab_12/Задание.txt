Внесенные мной изменения:

1) Применен шаблон проектирования MVP для разделения слоёв модели и отображения. 
В качестве view выступает WindowsForms форма PlotForm, в качестве модели классы решения уравнения.
В зависимости от выбора пользователя во время исполнения представитель/presentor
создаёт подходящий экземпляр Solver.

2) Для отложенного создания экземпляров ISolver применен паттерн "фабричный метод"

3)Изменения относительно пункта 2 задания:
"Создаём абстрактный класс AbstractSolver... Делаем реализацию метода SolveTheEquation: 
в цикле, пока модель разности A и B превышает Epsilon, 
вызываем метод OneStep и увеличиваем cчётчик итераций на единицу."

Метод SolveTheEquation() сделал виртуальным, в нём определил базовый фукнционал для всех
методов решения. Каждый наследник имеет свои реализации метода, в котором определяют
свои условия завершения цикла поиска решения и свой дополнительный функционал. Каждый 
наследник дополнительно обращается к базовому методу для выполнения общего функционала.

4)Изменения относительно пункта 3 задания:
"Создаём класс Dichotomia - наследник AbstractSolver, который имеет два закрытых вещественных поля
        private double fa; // значение Equation(a)
        private double fb; // значение Equation(b)
Создайте конструктор, обращающийся к конструктору базового класса и инициализирующий все поля. 
Он должен бросать исключение ArgumentException, если a>b или fa и fb имеют один знак."

Эту проверку перенёс в AbstractSolver, т.к. её будет проводить любой наследник 
(любой метод решения будет искать решения f(x) = 0 только если график функции пересекает ось x).


5) Изменения относительно пункта 8 задания:
"Добавьте ещё один класс - InformativeDichotomia (наследник Dichotomia). Он имеет открытое свойство
public Chart Plot { get; }
и конструктор
public InformativeDichotomia(Func<double, double> equation, double a, double b, Chart plot, double epsilon = 0.1e-6)
который обращается к методам базового класса и задает значения полей."

Во-первых нет смысла создавать наследника для каждого Solver с одинаковым функционалом,
поэтому функционал выдачи информации для построения графиков переезжает в базовый абстрактный класс.
Во-вторых ни модель, ни представитель не должны знать о способах отображения графика пользователю,
поэтому все данные из соответствующих функций будут передаваться наборами точек,
а представление уже будет решать задачу их отображения на своём графике.

6) Изменения относительно пунктов 6, 7, 8 задания:
Много изменений для соответствия модели MVP.
